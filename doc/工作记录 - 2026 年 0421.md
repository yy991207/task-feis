# 工作记录 - 2026 年 0421

## 本轮补充记录：任务详情补齐“关注的人”管理

### 改动背景

你这轮要求把任务详情底部的“关注的人”做成可用状态，不能再只是一个静态人数和加号占位。

现场排查后确认，原来的问题有两层：

- 详情页底部这块只有静态展示，没有真正接新增、展示、删除关注人的接口。
- 前端还把后端返回的 `participant_ids` 误当成了“额外负责人”，这会导致一旦新增关注人，负责人显示就会串掉。

### 主要修改点

#### 1. 把关注人数据链路从接口接到前端任务模型

- 文件：
  - `src/types/task.ts`
  - `src/services/taskService.ts`
- 处理方式：
  - 在前端 `Task` 类型里补上 `participant_ids`。
  - `apiTaskToTask` 转换时，把接口返回的 `participant_ids` 原样保留下来。
  - 同时把 `participant_ids` 和 `follower_ids` 都归到 `members` 的 `follower` 角色里，不再把 `participant_ids` 继续误映射成 `assignee`。
- 这样做的原因：
  - 详情页底部“关注的人”和主表“关注人”列都需要稳定的真实数据来源。
  - 先把语义对齐，后面的展示和交互才不会继续串字段。

#### 2. 详情页底部补齐关注人展示、添加、删除

- 文件：
  - `src/components/TaskDetailPanel/index.tsx`
  - `src/components/TaskDetailPanel/index.less`
- 处理方式：
  - 底部“关注的人”区域改成真实渲染当前关注人列表。
  - 增加“添加关注的人”弹层，使用 `POST /tasks/{task_id}/participants`。
  - 每个关注人右侧增加移除按钮，使用 `DELETE /tasks/{task_id}/participants/{target_user_id}`。
  - 每次新增或删除后，立即重新拉取任务详情并回写到页面层，保证详情页和主页面不用手动刷新就能看到最新状态。
- 这样做的原因：
  - 这是最小改动方案，直接复用现有任务详情刷新链路，不用新造状态同步机制。
  - 接口成功后再统一回写当前任务，可以避免本地拼装成员状态时漏字段。

#### 3. 负责人选择收口成单负责人，避免和关注人接口打架

- 文件：
  - `src/components/TaskDetailPanel/index.tsx`
  - `src/components/TaskTable/index.tsx`
- 处理方式：
  - 负责人选择改成单选，只走 `patchTaskAssignee`。
  - 任务清单里已有任务行、新建任务行、详情页负责人选择都统一成单负责人语义。
  - 不再通过 `participants` 接口去增删“额外负责人”。
- 这样做的原因：
  - 当前后端实际字段已经明确区分了 `assignee_id` 和 `participant_ids`。
  - 如果不把负责人入口一起收口，后续从主表改负责人时还是会把关注人数据带乱。

#### 4. 主表“关注人”列改成读真实关注人数

- 文件：`src/components/TaskTable/index.tsx`
- 处理方式：
  - 关注人列改成优先读取 `participant_ids`，再兜底旧的 `members.follower`。
  - 这样详情页改完关注人后，主表列里的关注人数会立即同步。
- 这样做的原因：
  - 之前主表只看 `members.follower`，详情页如果只改 `participant_ids`，主表会继续显示旧值。

### 异常处理手法

- 新增和删除关注人时，都统一用 `try/catch` 包住接口请求。
- 接口成功后重新拉任务详情，再统一回写页面层状态，避免本地只改了一半数据。
- 接口失败时直接提示“添加关注人失败”或“移除关注人失败”，不做静默吞错。

  - 待发送评论和已发送评论里的图片附件改成缩略图卡片展示，点击后进入统一的 `file-preview` 预览弹层。

## 根因说明

问题出在任务表格里的两段逻辑没有对齐：

1. 分组头部的加号按钮，点击后会把当前分组写进 `creatingInSection`。
2. 但真正负责把行内输入框渲染出来的代码，又额外要求 `config.toolbar.showCreate === true`。
3. 任务清单视图的默认配置里，`showCreate` 恰好是 `false`。

所以实际表现就是：

- 点击时状态已经改了；
- 但输入行不会显示；
- 用户看到的现象就是“点击无响应”。

## 异常处理手法

- 这次没有新增接口层异常处理。
- 修复重点放在前端渲染分支，避免把一个纯前端显示问题误判成接口问题。
- 回归测试覆盖的是“状态可进入，但 UI 必须能显示出来”这个关键链路，后面改动时能更早发现同类问题。
- 请求封装新增了空响应体兜底，避免 `DELETE`、`mark all read` 这类无返回内容的接口因为 `JSON.parse` 失败被误判成请求异常。
- 任务行删除父任务时，页面层会级联清理本地子任务，避免前端缓存和后端状态不一致。

## 本轮补充记录：任务清单表格重构为严格 Ant Design Table

### 改动背景

你这次明确要求把任务清单页里红框区域的表格改成严格的 Ant Design 组件，同时要保留当前已有功能，不能因为重构把分组、新建、字段配置、子任务展开这些链路打断。

原来的实现虽然已经用了不少 antd 基础控件，但表格本体还是手写列头、手写行、手写分组内容。这样后面继续加字段、调列宽、补交互时，维护成本会越来越高，也不利于和 antd 的视觉、语义、行为保持一致。

### 主要修改点

#### 1. 任务区主表格改成 `antd Table`

- 文件：
  - `src/components/TaskTable/index.tsx`
  - `src/components/TaskTable/index.less`
- 处理方式：
  - 直接引入 `antd/es/table`，把任务区渲染改成标准 `Table`。
  - 保留分组头部，改成“每个分组下面挂一张 Table”的结构。
  - 表头列定义统一收口到 `taskColumns`，不再额外手写一层 `.table-columns`。
- 这样做的原因：
  - 这次核心目标就是把表格收回到 antd 标准组件。
  - 分组头部继续单独保留，可以最小范围复用现有折叠、拖拽和分组管理能力。

#### 2. 把任务行能力拆成标准单元格组件

- 文件：`src/components/TaskTable/index.tsx`
- 处理方式：
  - 原来大而全的任务行渲染，拆成 `TaskTitleCell`、`TaskPriorityCell`、`TaskAssigneeCell`、`TaskDateCell`。
  - 标题列里继续保留：
    - 勾选完成状态；
    - 子任务展开箭头；
    - 任务标题重命名；
    - 显式“详情”按钮；
    - 子任务进度标签。
  - 优先级、负责人、开始时间、截止时间都改成 Table 列渲染，不再混在一整块手写行里。
- 这样做的原因：
  - 业务逻辑还能继续复用；
  - 表格结构已经切到 antd，但单元格交互不需要整套重写。

#### 3. 子任务展开继续保留，并接到 Table 的展开能力

- 文件：`src/components/TaskTable/index.tsx`
- 处理方式：
  - 顶层任务先转成 `TaskTableRow` 树形数据；
  - 已加载的子任务继续挂到 `subtasksByGuid`；
  - `expandedTaskGuids` 继续作为展开状态源；
  - `Table` 通过 `expandable` 接管展开行为，但隐藏默认展开列，继续用标题列里的箭头控制。
- 这样做的原因：
  - 保留原有的“只在展开时拉子任务”策略；
  - 避免一边用 antd Table，一边又额外套一层自己维护的树表结构。

#### 4. 自定义字段列继续支持编辑，但改成 Table 列渲染

- 文件：
  - `src/components/TaskTable/index.tsx`
  - `tests/custom-field-configuration-regression.test.mjs`
- 处理方式：
  - `CustomFieldCell` 保留；
  - 可见自定义字段不再在手写任务行里循环，而是在 `taskColumns` 里动态追加列；
  - 字段值更新继续走 `patchTaskCustomFields`，并保留本地乐观更新。
- 这样做的原因：
  - 你要求“保留当前功能的前提下重新设计表格”，自定义字段编辑是不能丢的核心能力。

#### 5. 视觉层改成 antd 表格风格，但保留当前产品的轻量样式

- 文件：`src/components/TaskTable/index.less`
- 处理方式：
  - 新增 `.task-grid-table`、`.task-table-row`、`.table-column-title`、`.table-empty-state` 等样式。
  - antd Table 的头部、行 hover、选中态、分组容器、横向滚动都做了定制。
  - 子任务树形引导线保留，没有因为 Table 重构丢掉层级感。
- 这样做的原因：
  - 如果只把结构换成默认 Table，页面会显得很生硬。
  - 这次是“严格 antd 组件”，不是“完全回退成 antd 默认皮肤”。

#### 6. 补回归测试，避免以后又回到手写表格

- 文件：
  - `tests/task-table-antd-regression.test.mjs`
  - `tests/custom-field-configuration-regression.test.mjs`
  - `tests/task-row-actions-regression.test.mjs`
  - `tests/subtask-indent-layout-regression.test.mjs`
- 处理方式：
  - 新增测试明确要求 `TaskTable` 直接引入并渲染 `antd Table`。
  - 原来几条绑定旧 DOM 结构的测试，一并改成匹配新表格实现，但继续覆盖：
    - 详情按钮仍然显式存在；
    - 子任务层级缩进仍然存在；
    - 自定义字段列仍然可编辑。
- 这样做的原因：
  - 这次是结构级重构，如果没有回归测试，很容易以后又被局部需求改回手写骨架。

### 异常处理手法

- 表格重构没有改接口契约，还是沿用原来的任务更新、字段更新、子任务加载和分组操作接口。
- 自定义字段、优先级、负责人、日期都继续保留本地乐观更新，接口失败时回滚并提示。
- 子任务开始时间继续保留只读保护，避免子任务单独改开始时间。
