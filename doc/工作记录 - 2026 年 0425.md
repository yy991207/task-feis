## 改动背景

根据接口调整说明，前端需要从原来的 team 维度分组接口切换到个人分组接口。核心变化是分组接口路径从 `project-groups` 切换到 `user-project-groups`，清单归属字段从 `group_id` 增加到 `user_group_id`，清单筛选参数也改为 `user_group_id`。

## 主要修改点

1. 分组接口服务调整
   - `GET /teams/{team_id}/project-groups` 改为 `GET /user-project-groups?user_id=xxx`。
   - `POST /teams/{team_id}/project-groups` 改为 `POST /user-project-groups`。
   - `PUT /project-groups/{group_id}` 改为 `PUT /user-project-groups/{group_id}`。
   - `DELETE /project-groups/{group_id}` 改为 `DELETE /user-project-groups/{group_id}`。
   - `PATCH /project-groups/{group_id}/sort-order` 改为 `PATCH /user-project-groups/{group_id}/sort-order`。

2. 清单接口服务调整
   - `listProjects` 的分组筛选参数从 `group_id` 改为 `user_group_id`。
   - 拖拽清单到分组的接口从 `PATCH /projects/{project_id}/group` 改为 `PATCH /projects/{project_id}/user-group`。
   - 拖拽 body 保持提交 `user_id` 和 `group_id`，与新接口说明一致。

3. 类型字段补齐
   - `ProjectGroup` 新增 `owner_user_id` 和 `project_count`。
   - `Project` 新增 `user_group_id` 和 `user_group_name`。

4. 侧栏分组数据适配
   - 新增 `getProjectGroupId`，优先读取 `user_group_id`，旧数据回退到 `group_id`。
   - 侧栏项目分组、未分组判断、拖拽同组判断和本地乐观更新都统一走这个方法。

## 异常处理手法

- 原有请求封装仍统一处理 HTTP 错误和业务错误码，这次没有改动错误处理入口。
- 拖拽分组失败时继续保留原有回滚逻辑：接口失败后恢复 `prevProjects`，并弹出错误提示。

## 补充记录：评论输入框粘贴多图后改成固定最大高度并内部滚动

### 改动背景

上一轮已经把评论区待发送图片预览挪进输入框内部了，但你这次反馈的新问题是：当图片越来越多时，整个输入框会继续被往下撑高，影响右侧详情区阅读。

这次要收口成参考图那种交互：

- 输入框保持一个默认最大高度；
- 图片和文本超过这个高度后，不再继续撑高外层；
- 通过输入框内部滚动查看剩余内容。

### 根因

- 当前评论输入态里，图片预览虽然已经进了 `.task-rich-input-shell`，但这个外层壳子还没有最大高度和纵向滚动约束。
- 结果就是附件列表每增加一张图，都会继续把 `.task-rich-input-shell` 的自然高度往下撑。
- 同时编辑区 `.task-rich-input-editor` 还保留着自己的 `max-height` 和 `overflow-y: auto`，如果再给外层直接加滚动，很容易变成双滚动条。

### 主要修改点

1. 给评论输入框壳子增加最大高度和内部滚动
   - 文件：`src/components/TaskRichInput/index.less`
   - 在 `.comment-input-wrapper .task-rich-input-shell` 下增加：
     - `max-height: 320px`
     - `overflow-y: auto`
   - 这样评论输入区整体在到达上限后会直接在壳子内部滚动。

2. 收掉评论编辑区自己的滚动
   - 文件：`src/components/TaskRichInput/index.less`
   - 把 `.comment-input-wrapper .task-rich-input-editor` 的：
     - `max-height: 180px` 改成 `max-height: none`
     - `overflow-y: auto` 改成 `overflow-y: visible`
   - 这样滚动责任统一交给外层壳子，避免文本区和图片区各滚各的。

3. 补充滚动约束回归测试
   - 文件：`tests/comment-input-inline-image-preview-regression.test.mjs`
   - 新增断言：评论输入框壳子必须有 `max-height: 320px` 和 `overflow-y: auto`。

## 补充记录：主表关注人列按单人和多人分开显示

### 改动背景

这轮你把主表“关注人”列的展示规则重新定清楚了：

- 只有 1 个关注人时，显示“头像 + 名字”；
- 有多个关注人时，参考负责人列，显示头像组。

也就是说，这一列不能再只显示人数，也不能把单人和多人都混成同一种文本展示。

### 根因

- 回滚后 `src/components/TaskTable/index.tsx` 里的 `followers` 列又回到了旧实现，还是本地统计 `followerCount` 后直接渲染数量。
- 当前文件其实已经有负责人列在用的头像样式、头像组样式，以及头像地址规范化和姓名兜底函数。
- 真正缺的是关注人列自己的用户整理逻辑，以及基于人数分支的渲染规则。

### 主要修改点

1. 新增关注人用户整理函数
   - 文件：`src/components/TaskTable/index.tsx`
   - 新增 `buildTaskFollowerUsers(task, users)`。
   - 统一从 `participant_ids` 和 `members.follower` 去重整理关注人。
   - 名字优先取用户表或成员名，头像走 `normalizeAvatarSrc` 统一兜底。

2. 主表关注人列改成双分支展示
   - 文件：`src/components/TaskTable/index.tsx`
   - 没有关注人时继续显示 `-`。
   - 单个关注人时，显示 20px 头像和右侧姓名。
   - 多个关注人时，复用负责人列同款 `Avatar.Group size={20} max={{ count: 3 }}` 头像组样式。

3. 补充单人展示样式
   - 文件：`src/components/TaskTable/index.less`
   - 新增 `.cell-followers-single` 和 `.followers-single-name`。
   - 保证单人时头像和姓名一行对齐，名字超出时省略，不把表格撑坏。

4. 回归测试改成新规则
   - 文件：`tests/task-table-followers-name-regression.test.mjs`
   - 新增断言：必须存在单人分支。
   - 新增断言：单人分支必须显示头像和名字。
   - 新增断言：多人分支必须使用 `Avatar.Group`。
   - 保留断言：不能再回退成 `followerCount` 人数统计。

### 异常处理手法

- 这次只改主表显示层，不新增接口，也不改现有错误处理入口。
- 关注人为空时继续显示 `-`，避免空白单元格。
- 头像为空时继续用现有首字兜底，避免出现破图或空头像。

### 涉及模块

- `src/components/TaskTable/index.tsx`
- `src/components/TaskTable/index.less`
- `tests/task-table-followers-name-regression.test.mjs`

### 可能影响范围

- 负责人列、参与人列、详情页关注人展示和关注人筛选逻辑这次都没有改。
