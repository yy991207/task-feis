# 工作记录 - 2026 年 0422

## 改动背景

这轮需求是把任务负责人入参从单人 `assignee_id` 改成多人 `assignee_ids`，同时兼容后端当前返回里还存在的旧字段，保证前端创建、更新、子任务和列表展示都能正常跑。

## 接口连通性

先按你给的真实接口格式做了连通性验证。

- `PATCH /api/v1/task-center/tasks/{task_id}/assignee` 已确认可接收 `assignee_ids`
- `POST /api/v1/task-center/tasks`、`PUT /api/v1/task-center/tasks/{task_id}`、`POST /api/v1/task-center/tasks/batch` 也一起做了兼容测试
- 当前后端返回里还是以 `participant_ids` 为主，`assignee_id` 仍可能存在，所以前端需要做兼容兜底

## 主要修改点

### 1. 任务服务层改成多人负责人入参

- 文件：`src/services/taskService.ts`
- 处理内容：
  - `ApiTask` 补了 `assignee_ids`
  - `createTaskApi`、`updateTaskApi`、`patchTaskAssignee`、`batchTasks` 改成支持 `assignee_ids`
  - `apiTaskToTask` 兼容 `assignee_ids`，同时兜底旧的 `assignee_id`
- 原因：
  - 前端不能继续只发单人负责人，不然和现在的接口格式对不上
  - 兼容旧字段是为了避免后端还没完全切完时页面直接失真

### 2. 负责人选择器改成可多选

- 文件：`src/components/UserSearchSelect/index.tsx`
- 处理内容：
  - 增加 `mode` 参数
  - `value` 和 `onChange` 支持单值和数组
- 原因：
  - 主表负责人、详情页负责人、子任务负责人都要复用同一个选人控件

### 3. 主表和详情页负责人改成多人展示与多人提交

- 文件：
  - `src/components/TaskTable/index.tsx`
  - `src/components/TaskDetailPanel/index.tsx`
- 处理内容：
  - 主任务创建时改发 `assignee_ids`
  - 主任务负责人列改成头像组展示
  - 详情页负责人改成多选
  - 子任务创建和子任务负责人编辑也一起改成多选
  - 提交后仍会同步关注人，避免本地展示断层
- 原因：
  - 这次改的是业务入参，不只是 UI 文案
  - 负责人和关注人不能混在一起，展示和提交都要分清楚

## 异常处理

- 负责人更新失败时，继续沿用原来的错误提示
- 创建子任务/任务失败时，保留现有兜底提示
- 关注人选择器保持原来的单选新增逻辑，不跟这轮负责人改造混在一起

## 可能影响范围

- 任务清单主表负责人列
- 任务详情页负责人行
- 子任务创建和子任务负责人编辑
- 批量更新负责人
- 依赖 `assignee_id` 的旧逻辑会逐步被 `assignee_ids` 取代
