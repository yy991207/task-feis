## 本轮补充记录：PERMISSION_DENIED 权限错误做前端可读反馈

### 改动背景

你给了一个典型接口返回：

```json
{
  "success": false,
  "code": "PERMISSION_DENIED",
  "msg": "仅任务创建人、负责人或团队管理员可执行此操作",
  "data": null
}
```

问题不在请求层，因为统一请求封装本来就会抛出后端 `msg`。真正的问题是前端很多任务操作在 `catch` 里直接写死了“更新失败”“状态更新失败”这类提示，把后端返回的权限原因盖掉了，所以用户看不到真实报错。

### 主要修改点

- 文件：
  - `src/components/TaskTable/index.tsx`
  - `src/components/TaskDetailPanel/index.tsx`
  - `tests/permission-denied-feedback-regression.test.mjs`
- 处理方式：
  - 在任务主视图和任务详情里增加统一的错误消息提取函数 `getActionErrorMessage`。
  - 任务状态切换、字段状态切换、自定义字段更新、详情页负责人更新、详情页当前任务状态切换、详情页子任务状态切换等操作，失败时优先展示 `err.message`。
  - 如果异常对象里拿不到后端错误信息，再回退到原来的通用提示文案。

### 异常处理手法

  - 请求层继续抛后端 `msg`
  - 主视图和详情页关键任务操作必须通过统一错误提取函数做反馈

## 本轮补充记录：侧栏 Tree 选中保护，避免空节点 scrollTo 报错

### 改动背景

你贴了两个报错，其中前端这条是：

```text
Uncaught TypeError: Cannot read properties of null (reading 'scrollTo')
```

堆栈落在 antd Tree 内部。排查后确认不是你业务代码里直接调了 `scrollTo`，而是侧栏 `Tree` 在受控 `selectedKeys` 模式下，拿到了一个当前 `treeData` 里还不存在的 key，内部尝试滚动定位激活节点时命中了空引用。

这类时机一般出现在：

- 当前激活清单 `activeKey` 已经切换

## 补充说明：Mongo `idx_involved` 索引冲突属于后端问题

### 现象说明

你贴的另一条错误是：

```text
Index must have unique name ... existing index name: "idx_involved"
```

这个报错发生在创建任务接口：

- 前端触发点：`src/components/TaskTable/index.tsx` 里的行内新建任务
- 请求入口：`src/services/taskService.ts` 里的 `createTaskApi`
- 最终失败位置：后端返回的 Mongo `IndexKeySpecsConflict`

### 根因判断

现有 Mongo 里已经有一个名字叫 `idx_involved` 的索引：

- 旧 key：`{ involved_user_ids: 1, is_deleted: 1, status: 1 }`

但当前后端又试图创建另一个同名索引：

- 新 key：`{ involved_user_ids: 1, is_deleted: 1, is_completed: 1 }`

Mongo 要求：

- 索引名必须唯一
- 同名索引的 key 规格也必须一致

所以这不是前端参数问题，而是后端索引迁移没有处理干净。

### 后端建议修法

后端二选一：

- 方案 1：给新索引换一个新名字，比如 `idx_involved_completed`
- 方案 2：如果确认要废弃旧索引，先显式删除旧的 `idx_involved`，再创建新的 `idx_involved`

如果后端是自动 `create_index` / `createIndex`：

- 不要在保留旧索引的情况下，直接复用原索引名去创建不同 key 的索引
- 最好给索引初始化加幂等迁移逻辑，先检查已有索引名和 key 是否一致，再决定跳过、删旧重建，还是新建新名字

### 当前处理边界

- 这个仓库里没有后端索引初始化代码，所以这轮没有在本仓库里直接修 `idx_involved`
- 前端这边保留真实后端报错透出，不会把它吞成模糊的“创建失败”

## 本轮补充记录：任务完成态改为读取接口 `is_completed`

### 改动背景

你说明了任务状态切换接口的新返回结构，当前任务是否完成不再靠旧的 `status` 字段判断，而是明确通过：

```json
"is_completed": true
```

来表示当前任务已完成。前端现状是：

- 勾选框和完成态样式仍然统一读 `task.status === 'done'`
- 但 `task.status` 的来源还是 `apiTaskToTask(api.status)`

这会导致一个问题：

- 如果后端已经切到 `is_completed`
- 但 `status` 没再返回旧的 `done`
- 前端映射出来的 `task.status` 就可能不是 `done`
- 最终表现为任务名前面的方框勾选状态不对

### 主要修改点

- 文件：
  - `src/services/taskService.ts`
  - `tests/task-completion-flag-regression.test.mjs`
  - `tests/custom-field-api-integration-regression.test.mjs`
- 处理方式：
  - `ApiTask` 模型增加 `is_completed: boolean`
  - `normalizeTaskStatus` 改成同时接收 `status` 和 `is_completed`
  - 归一化逻辑里优先判断：
    - `is_completed === true` 时，直接映射为前端 `Task.status = 'done'`
  - `apiTaskToTask` 改为：
    - `status: normalizeTaskStatus(api.status, api.is_completed)`
  - 主视图、详情页和活动流里的勾选框逻辑不额外分叉，继续读取映射后的 `task.status === 'done'`，这样改动最小，也不会把 UI 逻辑搞成两套判断
  - 同步更新原有状态相关回归测试，允许状态归一化函数读取新的 `is_completed`

### 异常处理手法

- 不去大面积改所有 UI 里的完成态判断，而是把兼容统一收口到服务层映射，避免同一语义在多个组件里散着判断。
- 保留旧的 `status` 兼容：
  - 如果 `is_completed` 为 `false`，仍然继续读 `status` 里的 `in_progress`、`cancelled` 等真实状态值。
